# iOS / Android フロントエンド差分表

## 目的

この文書は、SwiftUI 版 iOS フロントエンドと Flutter 版 Android フロントエンドを細部まで一致させるための差分整理表です。

- 基準 UI: iOS / SwiftUI 実装
- 対象 UI: Flutter 実装
- 差分の分類:
  - 機能差: 片方にしか存在しない、または操作フロー自体が違う
  - 画面設計差: 同じ機能でもレイアウトや導線が違う
  - 見た目差: 色、余白、フォント、ボタン形状、モーダル表現などの差

## 差分サマリー

| 区分 | 内容 | 影響度 | 優先度 |
|---|---|---|---|
| 機能差 | タイマー画面、起動スプラッシュ、投了 UI、フォルダ管理、広告、振り駒 UI 接続 | 高 | 高 |

### 1-10. URL 登録 / ユーザー登録 / backend 設定

| 項目 | iOS | Flutter | 差分種別 | 統一方針 |
|---|---|---|---|---|
| URL 登録 | 独立シート | 単独利用時は独立、ホーム導線では統合 | 画面設計差 | 導線を統一 |
| backend URL 入力 | backend 設定シートに分離 | URL 登録シート内でも入力可能 | 画面設計差 | iOS に寄せるか統合方針に寄せる |
| ユーザー名登録 | 独立シート | URL 登録シート内でも可能 | 画面設計差 | 分離 / 統合の方針統一 |

重要:

- ここは Flutter 側の方が機能を集約している。
- iOS 準拠で揃えるなら Flutter を分割する必要がある。
- 逆に Flutter 方式を採用するなら iOS 側も再設計が必要。

## 2. 画面設計差分一覧

### 2-1. ルート構造

| 観点 | iOS | Flutter | 統一時の論点 |
|---|---|---|---|
| 画面遷移 | 単一 `ContentView` 内モード切替 | `HomePage` → `GamePage` push | 単一シェルか画面遷移型かを決める |
| シート管理 | ルートで多くを制御 | 各画面から bottom sheet を開く | sheet の責務分担を揃える |
| 状態管理 | `@StateObject` 集中 | Riverpod controller 分散 | UI 構造と state 所有者の境界を揃える |

### 2-2. 保存棋譜画面の構造

| 観点 | iOS | Flutter | 統一時の論点 |
|---|---|---|---|
| ベースコンポーネント | `List + Section` | `ListView + Card` | 情報管理画面をどちらに寄せるか |
| 操作ボタン | toolbar に少数配置 | header に多数 icon 配置 | アクション密度を揃える |
| 情報表現 | テキスト中心 | カード中心 | 管理画面としての視認性を優先する |

### 2-3. フォーム画面の構造

| 観点 | iOS | Flutter | 統一時の論点 |
|---|---|---|---|
| URL 登録 | 単機能シート | 複合シート化可能 | 単機能主義か統合主義か |
| ユーザー登録 | 独立 | 独立と統合の両方 | 再利用コンポーネント設計を揃える |
| backend 設定 | 独立 | 独立だが URL 登録でも入力可 | 冗長導線の扱いを決める |

## 3. 見た目差分一覧

## 4. 優先度付き実装ギャップ

### 優先度 A: 先に埋めないと「同じフロントエンド」にならない差分

| 項目 | 内容 |
|---|---|
| 独立タイマー | Flutter に存在しない |
| 振り駒 UI | Flutter は state のみで画面未実装 |
| 投了 UI | Flutter に対局中の投了導線がない |
| 保存棋譜フォルダ管理 | Flutter にない |
| 対局前設定の入力モデル | picker 方式と preset 方式で異なる |
| URL / backend / user の導線構造 | iOS と Flutter で分離 / 統合方針が異なる |

### 優先度 B: 機能は近いが設計差が大きい項目

| 項目 | 内容 |
|---|---|
| 対局画面の密度 | Flutter がコンパクト寄り |
| 保存棋譜一覧の UI 骨格 | iOS はリスト、Flutter はカード |
| 終局ポップアップ | ボタン階層と閉じ方が違う |
| レビューコントロール | Material 依存で見た目が違う |

### 優先度 C: 最終調整レベルの差分

| 項目 | 内容 |
|---|---|
| 色の細差 | shadow, overlay, warning など |
| フォント | 和文見出しや演出文字 |
| 余白 | section 間隔、カード余白、シート角丸 |
| アイコン | SF Symbols と Material Icons の差 |

## 5. Flutter を iOS に揃える場合の実装順

1. 独立タイマー画面を追加する
2. 振り駒演出を `GamePage` に接続する
3. 投了導線を追加する
4. 保存棋譜のフォルダ管理とフィルタを追加する
5. 対局前設定を iOS と同じ入力モデルに揃える
6. URL 登録、ユーザー登録、backend 設定を iOS と同じ分離構造に揃える
7. 終局ポップアップ、検討 UI、ホーム画面の余白とボタンスタイルを調整する
8. スプラッシュと広告対応を必要に応じて追加する

## 6. 結論

現状の Flutter フロントエンドは、iOS と色調や主要機能の方向性は近いものの、以下の理由で「同じ UI」とは言えない状態です。

- 未実装の機能差が複数ある
- 入力フロー自体が異なる画面がある
- 一覧画面と対局画面の情報設計が異なる
- モーダル、ボタン、余白、文字組みのルールが統一されていない

最小限の見た目合わせではなく、本当に統一するなら、Flutter 側を iOS の画面仕様に寄せて再構築するのが最短です。
